# 模块交付标准

> 目标：确保每个模块都可以**独立设计、独立实现、独立中断、独立恢复、独立验收**。关联：[MODULE_DESIGN.md](./MODULE_DESIGN.md) · [MODULE_INTERFACES.md](./MODULE_INTERFACES.md) · [CODING_STANDARDS.md](./CODING_STANDARDS.md) · [ROADMAP.md](./ROADMAP.md)

## 1. 为什么需要这个标准

当前仓库已经有：

- 总体架构设计
- 单模块详细设计
- 模块接口契约
- 编码规范

但如果一次开发中途异常中断，单靠“设计文档”还不够。真正能保证恢复速度和可控性的，是每个模块都有一套完整的交付资料，至少覆盖：

- 设计图
- 设计思路
- 实现过程记录
- 验证结果
- 使用与维护说明

本标准就是把这些内容固定成统一要求。

## 2. 每个模块必须具备的文档包

每个模块最终至少要有以下 5 类资料：

| 类别 | 作用 | 是否必需 |
|------|------|----------|
| 1. 设计文档 | 定义模块做什么、边界、接口、数据结构 | 必需 |
| 2. 设计图 | 用图说明流程、依赖、状态机、时序 | 必需 |
| 3. 实施记录 | 记录怎么落地、做到哪一步、变更过什么 | 必需 |
| 4. 验证记录 | 记录测试、验收、已知缺陷、风险 | 必需 |
| 5. 使用说明 | 告诉后续开发者怎么运行、怎么排错、怎么扩展 | 必需 |

换句话说，单模块不能只有“想法”，必须形成一套可恢复的工作包。

## 3. 推荐目录结构

建议每个模块在 `docs/modules/` 下同时维护一组配套资料。

```text
docs/modules/
  01-domain.md                 # 设计文档（已存在）
  01-domain-implementation.md  # 实施记录
  01-domain-verification.md    # 验证与验收
  01-domain-operations.md      # 使用/运维/排错说明
```

如果模块流程复杂，可以额外拆分图示文档：

```text
docs/modules/assets/
  01-domain-flow.mmd
  06-surveillance-sequence.mmd
```

也可以继续把图直接内嵌在主文档中；重点不是文件数量，而是信息齐全。

## 4. 五类资料的最小内容要求

### 4.1 设计文档

现有 `docs/modules/*.md` 基本已经覆盖这一层，至少应包含：

- 模块定位与边界
- 目录与文件结构
- 核心数据结构与类
- 核心流程
- 对外接口契约
- 配置与表
- 错误处理与降级
- 测试用例清单
- 开放问题

### 4.2 设计图

每个模块至少要有 1 张高价值图，复杂模块建议 2~3 张。常见类型：

- 流程图：模块主流程
- 时序图：模块与上下游交互
- 状态机图：任务状态 / 告警状态 / 生命周期
- 依赖图：模块内部组件关系
- 数据流图：输入、转换、输出

最低要求不是“画得漂亮”，而是别人中断后 5 分钟能看懂主链路。

### 4.3 实施记录

实施记录不是流水账，而是帮助恢复现场的最小事实集。至少要记录：

- 当前实现目标
- 本次范围
- 已完成项
- 未完成项
- 关键决策
- 实际代码路径
- 依赖的配置、表、环境
- 与原设计偏离的地方
- 下一步该做什么

### 4.4 验证记录

验证记录至少要能回答这几个问题：

- 这个模块测过什么
- 哪些场景已通过
- 哪些边界还没覆盖
- 当前剩余风险是什么
- 是否达到本阶段验收

不能只写“已测试通过”，必须列出具体验证项。

### 4.5 使用说明

模块级使用说明至少要写：

- 如何启动 / 调用
- 关键配置项
- 输入输出示例
- 常见错误与排查步骤
- 如何扩展
- 改动这个模块时需要注意什么上游/下游影响

## 5. 模块开发的标准工作流

每个模块都按下面顺序推进，避免直接跳到写代码：

1. 先确认设计文档已完整
2. 补足设计图，确保主流程可视化
3. 开始实现前创建实施记录文件
4. 编码过程中持续更新实施记录
5. 模块完成后补验证记录
6. 最后整理使用说明

这意味着“模块完成”的定义不只是代码完成，而是**代码 + 文档包完成**。

## 6. 中断恢复机制

为了应对一次开发中途中断，每个模块的实施记录必须支持下面三件事：

### 6.1 看一眼知道做到哪

实施记录顶部必须有状态区，例如：

```text
状态：进行中 / 已完成 / 阻塞
当前阶段：schema 已完成，service 未完成
最近更新时间：2026-06-22
```

### 6.2 看一眼知道差什么

必须明确写出：

- 当前阻塞点
- 缺什么依赖
- 哪些测试没补
- 哪些接口还未对接

### 6.3 看一眼知道下一步怎么接

必须有“下一步动作”清单，按执行顺序写，不要只写目标。

## 7. 与现有文档的关系

这份标准不替代现有设计文档，而是在其上补足“交付过程资料”：

| 已有文档 | 作用 |
|----------|------|
| `MODULE_DESIGN.md` | 全局模块边界 |
| `docs/modules/*.md` | 单模块设计说明书 |
| `MODULE_INTERFACES.md` | 模块之间怎么对接 |
| `CODING_STANDARDS.md` | 编码与工程约束 |
| 本文档 | 模块如何形成可恢复、可验收的交付包 |

## 8. 建议的状态标签

为了统一跟踪，模块实施记录建议使用固定状态：

- `planned`：已设计，未开工
- `in_progress`：实现中
- `blocked`：被依赖或外部条件阻塞
- `implemented`：代码完成，待验证
- `verified`：验证完成
- `documented`：说明文档完成
- `done`：达到本阶段交付

## 9. 最低执行要求

从现在开始，任何进入编码阶段的模块，至少要同步产出：

1. 原有设计文档
2. `*-implementation.md`
3. `*-verification.md`
4. `*-operations.md`

如果模块尚未编码，可以先只保留设计文档；一旦开工，就必须把后三类补上。

## 10. 推荐做法

首轮实现建议不要一次性给 14 个模块都补完整交付包，而是按 Phase 推进：

- Phase A 开工时，先给 `domain / config / storage / providers / ingest / api / frontend / observability` 建立配套模板
- 后续模块再按同样规范复制

这样做的目的不是增加文档负担，而是降低中断成本、交接成本和返工成本。
